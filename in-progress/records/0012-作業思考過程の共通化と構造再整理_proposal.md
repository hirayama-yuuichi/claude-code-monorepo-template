## 0012: 作業・思考過程の共通化、及び構造の再整理

- 日付: 2026-04-27
- 状態: 議論中
- ドキュメント: 未着手
- 実装: 未着手

### 背景

作業・思考過程の共通化のルールが曖昧なのではっきりしたい。
具体的には「アイデアをどこに書くか」「いつ records 化するか」「backlog とどう使い分けるか」が人によって（あるいは Claude と人間の間で）ブレていた。

### 指摘

- アイデアを ideas.md に書くべきか backlog.md に書くべきか迷う
- records を作るタイミングが不明確
- 作業完了後に何をどう更新すればよいか共通化されていない

### 判断

`in-progress/` に完成品以外（判断・進捗）をすべて集約する。

`in-progress/` に置くもの：

- `plan/` — パッケージ横断の計画（Why/What/How）。人間・Claude が管理
- `ideas.md` — まだやるか不明なアイデアの一元管理。人間が管理
- `records/` — 設計判断ログ。採用後もここに置いたままステータスで追跡。Claude が管理（人間も参照）
- `backlog.md` — 作業リスト（優先度順）。人間が管理

各ファイルの記載方法：

- `plan/` — Why/What/How の3段構成で書く。パッケージをまたぐ計画のみ記載し、個別パッケージの詳細は書かない
- `ideas.md` — 見出し1行のみ。詳細は書かない。やると決まったら `/records-add` で records 化する
- `records/` — `/records-add` スキルで作成。背景・判断・理由・却下した選択肢・実装方針を記載。ステータスフィールドで進捗を追跡
- `backlog.md` — 見出し1行のみ（詳細は records で検討）。優先度順に並べる。完了したら `YYYY/MM/DD` 見出し下に「済」付きで移動

作業フロー：

```
plan/ に全体の計画を作る
    ↓
必要があれば ideas.md にアイデアを書く
    ↓ やると決まったら
/records-add で records 化 → backlog.md にも追加
    ↓
実装・docs 更新 → record のステータスを完了に更新
    ↓
backlog.md も済にする
```

判断に迷うときは批判と修正案で迷いを取り除く：

```
/critique → /propose → 採用
```

### 理由

- 「どこに書くか迷わない」ための一元管理。書き先が明確だと作業の摩擦が減る
- 管理主体を分けることで、Claude と人間の役割が明確になる
- フローを明示することで、アイデアが実装まで辿り着く経路が共通化される

### 却下した選択肢

| 案 | 却下理由 |
|---|---|
| ideas と backlog を1ファイルにまとめる | 「まだやらない」と「やる」が混在して優先度が不明確になる |
| records を docs/ に置く | 議論中・採用済みが混在するため「完成情報のみ」の docs/ とは合わない |

### 影響

- CLAUDE.md の出力ルール表がこの構造を反映する必要がある
- 各ファイルの役割が明確になったため、書き先の迷いが解消される
