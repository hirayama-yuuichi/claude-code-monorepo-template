# 設計判断ログ

## 0015: backlog の役割と用途の定義

- 日付: 2026-04-28
- 状態: 採用
- ドキュメント: 完了
- 実装: 完了

### 背景

backlog なんとなく作ったが、イマイチ、役割が明確ではない。
役割を確認したい。

### 判断

役割としては：
- 作業の漏れを防止する
- 作業順番を確認する

管理主体の分担：
- **records ステータスフィールド**：Claude が管理する設計判断の反映状態（実装・docs の完了追跡）
- **backlog**：人間が管理する作業リスト。records に収まらない雑多なタスクも含めてよい

### 理由

- タスクを頭の中ではなく外部に書き出すことで、忘れや抜けが起きない
- 優先度順に並べることで「次に何をやるか」が即座にわかる
- records ステータスと backlog は「二重管理」ではなく「管理主体が違う」ため矛盾しない

### 却下した選択肢

| 案 | 却下理由 |
|---|---|
| records ステータスフィールドで作業管理を兼ねる | records は設計判断の記録が目的で、作業の順番管理には向かない |
| アイデアと一緒に管理する（ideas.md） | やると決まっていないものと混在して優先度が不明確になる |
| backlog を直近キューに限定する | 優先度順で並んでいれば全量書いても埋もれない。管理主体の分担で十分 |

### 影響

- CLAUDE.md の `やること追加` 欄の説明をこの定義に合わせて更新する

### docs 記載方針

不要（CLAUDE.md と backlog.md のコメントで対応）

### 実装方針

- `in-progress/backlog.md` の冒頭コメントを以下に変更：

