## 0011: モノレポの共通化、及び構造の再整理

### 背景

rec-0010 で整理した「やりたいこと」の1つ目。現状の `packages/template/` 構造を確認し、展開先プロジェクトで即日機能する状態にするための不足・ズレを明らかにする。

### 現状の構造（template/）

```
packages/template/
├── README.md                        ← リポジトリ入口。プロジェクト概要・パッケージ一覧・最短の使い方
├── docs/
│   ├── architecture.md              ← 全体構成・パッケージ間依存・責務分離方針
│   ├── setup.md                     ← 環境構築手順（新人がここだけ見れば動く状態を目指す）
│   └── records/
│       └── 0001-xxx.md              ← 設計判断ログ（ADR）のプレースホルダー
├── ideas/
│   └── ideas.md                     ← まだやるか不明なアイデアの一元管理
├── tasks/
│   ├── backlog.md                   ← やること一覧（唯一のTODO。ここだけに書く）
│   └── plan.md                      ← パッケージ横断の計画（Why/What/How）
└── packages/
    ├── package-a/
    │   ├── README.md
    │   ├── plan.md
    │   ├── setup.md
    │   └── spec/
    │       ├── api.md
    │       ├── data.md
    │       └── rules.md
    └── package-b/
        └── .gitkeep
```

### 現状の指摘

- `CLAUDE.md` がない → 展開先プロジェクトで Claude Code が出力ルール・採用ループを知れない
- `docs/records/0001-xxx.md` がプレースホルダーのみ → ADR の書き方・用途が伝わらない
- `tasks/plan.md` はあるがルートの `plan.md` はルート直下にあり非対称（rec-0009）
- `docs/setup.md` の内容が空に近く、展開後に使えない状態
- `ideas/` と `tasks/` の2ディレクトリに「進行中の情報」が分散している

### 判断

以下の構造に変更する。

**コンセプト：完成・確定したものと、進行中のものを2ゾーンに分ける**

| ゾーン | 中身 |
|---|---|
| `in-progress/` | 進行中のすべて（アイデア・backlog・計画・ADR） |
| `docs/` | 完成・確定した情報（ADRの決定を反映済みのもの） |
| `packages/` | 実装 |

```
packages/template/
├── README.md                        ← リポジトリ入口
├── CLAUDE.md                        ← Claude Code 向け出力ルール・スキル一覧・採用ループ
├── docs/
│   ├── architecture.md              ← 全体構成・パッケージ間依存・責務分離方針
│   └── setup.md                     ← 環境構築手順
├── in-progress/
│   ├── ideas.md                     ← まだやるか不明なアイデアの一元管理
│   ├── backlog.md                   ← やること一覧（唯一のTODO。ここだけに書く）
│   ├── plan/
│   │   ├── plan.md                  ← パッケージ横断の計画（Why/What/How）
│   │   └── package-a.md             ← package-a の計画（Why/What/How）
│   └── records/
│       └── 0001-xxx.md              ← 設計判断ログ（ADR）
└── packages/
    ├── package-a/
    │   ├── CLAUDE.md                ← このパッケージ固有の Claude Code 向け指示
    │   ├── README.md                ← このパッケージの入口
    │   ├── src/                     ← ソース
    │   └── docs/
    │       ├── setup.md             ← このパッケージ環境構築手順
    │       ├── api.md               ← API仕様
    │       ├── data.md              ← データ構造・スキーマ
    │       └── rules.md             ← ビジネスルール・制約
    └── package-b/
        └── .gitkeep
```

**ADR のフォーマットに追加するステータスフィールド：**

```markdown
- 状態: 提案 | 議論中 | 採用 | 廃止
- ドキュメント: 未着手 | 作業中 | 完了
- 実装: 未着手 | 作業中 | 完了
```

ADR が採用されたとき、ドキュメントと実装のステータスをテキストで追跡する。両方「完了」になった状態が「完全に反映済み」となる。

### 理由

- `ideas/` と `tasks/` の2分割は「進行中の情報が2か所に分散する」という問題を生んでいた。`in-progress/` に一本化することで書き先の迷いがなくなる
- `in-progress/` という命名は開発者以外にも意味が伝わり（「進行中」）、`ideas/` よりも ADR や backlog を含む場所として自然
- `records/` を `in-progress/` 配下に置くのは、ADR が「確定後も in-progress/ にある」という状態になるが、チェックボックスで反映状態を追跡することで補う
- ルートに `CLAUDE.md` を追加することで、展開先プロジェクトで Claude Code が即日機能する

### 却下した選択肢

| 案 | 却下理由 |
|---|---|
| `ideas/` のまま（命名変更なし） | ADR・backlog を含む場所として「アイデア」という名前は不適切 |
| `tasks/` と `ideas/` の2分割を維持 | 情報が2か所に分散し、書き先の迷いが生じる |
| `records/` を `docs/` 配下に残す | ADR は議論中と採用済みが混在するため、「完成情報のみ」の `docs/` とは合わない |
| `wip/` という命名 | 開発者以外に意味が伝わりにくい |

### 影響

- `tasks/` が `in-progress/` に統合されるため、`CLAUDE.md` の出力ルール（`tasks/backlog.md` への書き込みルール）を `in-progress/backlog.md` に更新する必要がある
- 既存の展開済みプロジェクトがあれば、ディレクトリ名の変更が必要になる
- ADR フォーマットにチェックボックスを追加するため、既存の `docs/records/` ファイルも更新が必要

### docs 記載方針

- `CLAUDE.md`（ルート・template）の出力ルール表を `in-progress/` パスに更新
- `packages/template/CLAUDE.md` を新規作成
- `packages/template/packages/package-a/CLAUDE.md` を新規作成
- スキル（`records-add`・`critique`・`propose`）の参照パスを更新

### 実装方針

- `tasks/`・`ideas/` を `in-progress/` に統合（git mv）
- `docs/records/` を `in-progress/records/` に移動（git mv）
- `plan.md`（ルート直下）を `in-progress/plan/plan.md` に移動
- `packages/template/packages/package-a/spec/` を `docs/` に統合
- `packages/template/packages/package-a/src/` を追加
- 全 ADR にステータスフィールドを追加
