# 設計判断ログ

## 0001: Claude 提案の採用ループ

- 日付: 2026-04-25
- 状態: 採用
- [ ] docs/ 更新済み
- [ ] 実装済み

### 背景

Claude Code を使って計画・設計を進める際、Claude の提案を受け取ると情報量が多すぎて整理できない。
そのまま採用してしまうと根拠が残らず、誤った前提に気づけないまま計画が進む。

### 判断

Claude 提案を採用する際は以下のループを踏む。

```
1. Claude が提案する（元ファイルに反映 or チャットで提示）

2. 別インスタンスで批判する
   → 批判結果を「<対象>_review.md」として保存

3. 批判をもとに Claude が修正案を作成する
   → 修正内容を「<対象>_proposal.md」として保存

4. 元ファイルと変更案ファイルを並べて比較し、採用・却下を判断する
   → IDE で両ファイルを開いて差分を目視確認

5. 採用する場合 → 元ファイルを変更案の内容で更新する
   却下する場合 → 却下理由を Claude に伝えて 1. に戻る
```

### ファイル命名規則

| ファイル | 役割 |
|---|---|
| `<対象>.md` | 元ファイル（正本） |
| `<対象>_review.md` | 批判ファイル（別インスタンスが出力） |
| `<対象>_proposal.md` | 提案ファイル（批判をもとに Claude が作成） |

例: `plan.md` / `plan_review.md` / `plan_proposal.md`

### 理由

- 元ファイル・批判ファイル・変更案ファイルを分けることで、比較しながら採用判断できる
- 情報量が多くてもファイル単位で整理されるため、一度に判断する量を絞れる
- 批判フェーズを別インスタンスに委ねることで、提案者バイアスを排除できる
- 採用根拠がファイルとして残り、後から見直せる

### 却下した選択肢

| 案 | 却下した理由 |
|---|---|
| Claude 提案をそのまま元ファイルに書く | 根拠が残らず、誤った前提に気づけない |
| チャット内だけで比較・判断する | 情報量が多く整理できない |
| 批判と変更案を1ファイルにまとめる | 比較しにくく、元ファイルとの差分が追いにくい |

### 影響

- 重要な提案を受けるときは必ずこのループを経る
- 批判フェーズは別インスタンス（サブエージェント）で実施する
- `_review.md` と `_proposal.md` は一時ファイルとして扱い、採用後は削除または `_old/` に移動する
