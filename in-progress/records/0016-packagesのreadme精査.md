## 0016: packages/ の readme 精査

### 背景

rec-0016 の当初の意図は `packages/readme.md` の精査だったが、実際に精査すべきは `packages/template/packages/package-a/README.md`（テンプレートとして配布されるパッケージの README）であることが判明。

このファイルは `/package-add` でパッケージを追加したときに展開される。汎用的な内容だが、実際のプロジェクトではパッケージ固有の内容に書き直す必要がある。

### 判断

`packages/template/packages/package-a/README.md` を最小限の構造に変更する。

内容：
```markdown
# package-a

> このパッケージの役割を一言で記載してください

## 関連ドキュメント

- [CLAUDE.md](./CLAUDE.md) — このパッケージの開発方針
- [docs/](./docs/) — 仕様・セットアップ
```

### 理由

**不要と判断する理由：**

1. **CLAUDE.md と役割が重複**
   - `package-a/CLAUDE.md` がパッケージ固有の Claude 向け指示を提供している
   - README.md は人間向けだが、テンプレートとしては汎用的すぎて実用性が低い

2. **プレースホルダが少ない**
   - 実際のパッケージ名や目的は展開後に人間が書き直す必要がある
   - テンプレートとして提供する価値が薄い

3. **ディレクトリ構造で十分**
   - `src/`, `docs/` があれば、パッケージの役割は推測できる
   - README なしでも CLAUDE.md があれば Claude は動作する

4. **メンテナンス負担**
   - テンプレートに含めると、内容の一貫性を保つ必要がある
   - プロジェクトごとに異なるため、汎用的な内容を維持するのが難しい

5. **実際のパッケージ作成時に Claude が生成できる**
   - CLAUDE.md に指示すれば、プロジェクト固有の README を Claude が適切に生成できる

**最小構成（案A）を採用する理由：**

1. **GitHub での視認性を保つ**
   - パッケージを開いたときに説明が見える
   - 最低限の構造だけ提供し、詳細は CLAUDE.md と docs/ に委譲

2. **保守負担を最小化**
   - 汎用的な内容を維持する必要がない
   - プロジェクト固有の内容は展開後に書く

3. **足場としての役割**
   - 「何を書くべきか」の方向性だけ示す

### 却下した選択肢

| 案 | 却下理由 |
|---|---|
| README.md を削除する | GitHub での視認性が失われる。パッケージの説明場所がなくなる |
| 現在の構造を維持する | 汎用的な内容では実用性が低く、展開後に書き直す手間が発生する |
| プレースホルダを追加して自動置換する | パッケージの説明は機械的に置換できる内容ではない |

### 影響

- `packages/template/packages/package-a/README.md` の内容が最小構成に変更される
- `/package-add` スキルは影響なし（README の内容を前提としていない）
- パッケージ展開後、必要に応じて説明を追記する運用になる

### docs 記載方針

不要（テンプレート構造の変更のみ）

### 実装方針

`packages/template/packages/package-a/README.md` を最小構成に書き換える
