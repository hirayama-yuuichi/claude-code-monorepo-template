# plan.md レビュー結果

### 批判1: `/template-sync` が解決不可能な問題を抱えており、Why に答えていない

**レビュー結果**: Why の第1課題「ディレクトリ構造がバラバラ」を解消するのが `/template-sync` の役割だが、スキル詳細に書かれた「既存ファイルは上書きせずマージ候補として提示する」という方針が根本的に破綻している。テンプレートと既存プロジェクトの差分が生じた場合、「マージ候補として提示する」だけでは人間が手動でマージする必要があり、コマンド1つで均一構造を維持するという What を達成できない。特に CLAUDE.md や `docs/decisions/` のような書き込み済みファイルが対象になったとき、「提示するだけで適用しない」なら同期の意味がない。また「差分を表示し確認後に適用」と書かれているが、何を比較の基準とするか（どのバージョンのテンプレートと差分を取るか）が未定義であり、実装者は実装できない。

**改善案**: 同期対象を「構造のみ（ディレクトリ・空ファイル）」と「内容も含むファイル」に明示的に分類し、前者は自動上書き、後者はスキップしてレポート出力するという振る舞いを定義する。加えてテンプレートのバージョン管理方法（タグ？コミットハッシュ？）と、プロジェクト側でどのバージョンを適用済みかを追跡する仕組みを計画に含める必要がある。

---

### 批判2: 「対話的に置換」が Claude Code スキルとして成立しない

**レビュー結果**: `/monorepo-init` と `/package-add` のどちらも「プレースホルダを対話的に置換」と書かれているが、Claude Code のスキルはチャット上で動作するため、「対話的な入力フォーム」が存在しない。Claude がチャットで質問し、ユーザーが返答し、Claude がファイルを書き換えるというフローになるが、この場合にプレースホルダの定義・形式（`{{PROJECT_NAME}}` など）、置換対象ファイルの範囲、置換失敗時の挙動がいっさい記載されていない。`degit` を使うと明記されているが、degit の出力はローカルにそのまま展開されるため、プレースホルダが残ったままになる。Claude がどのファイルのどの文字列を書き換えるかのロジックは、計画を読んだだけでは実装できない。

**改善案**: プレースホルダの形式（例: `__PROJECT_NAME__`）を定義し、置換対象ファイルのリスト（あるいは全ファイル対象のグロブパターン）を明記する。「対話的に」の意味を「Claude がチャットで質問し、返答を受けてから書き換える」と明文化し、質問項目（プロジェクト名、パッケージ名、説明文など）を具体的にリストアップする。

### 批判3: テスト方法が「差分ゼロ」という検証不能な条件に依存している

**レビュー結果**: How-1・How-4 のテスト PASS 条件がいずれも「`モノレポ案.md` の仕様と差分ゼロ」となっているが、`モノレポ案.md` はディレクトリ構造のメモであり、機械的に比較できる形式ではない。「差分ゼロ」を実際に確認するコマンドも示されていない。How-2 の「差分が意図した変更のみ」も同様で、「意図した変更」を誰がどう判断するかが未定義であり、再現性のあるテストにならない。

**改善案**: `root/` の現在のディレクトリ・ファイル構成をテストの期待値として `find root/ -type f | sort` の出力でスナップショット化し、それとの diff を確認するコマンドを PASS 条件に書く。または `test/expected-structure.txt` のような期待ファイルを作成してテスト可能にすることを計画に含める。

---

### 批判4: スキルのグローバル配置リスクが未検討

**レビュー結果**: 「各スキルはグローバルに配置し、どのプロジェクトでも共用できる」と書かれているが、グローバル配置に伴うリスクへの言及がない。特に `/monorepo-init` は「degit hirayama-yuuichi/claude-code-monorepo-template/root」というハードコードされたリポジトリ名が入っており、リポジトリ名を変えたときやフォークして使うときに壊れる。スキル間の依存関係（`/propose` は `/critique` の出力を前提とする）も計画に明示されていない。

**改善案**: グローバルスキルが持ってよい前提（パス、リポジトリ名）と持ってはいけない前提を整理する。ハードコードされたリポジトリ URL は設定ファイルから読み込む方式にする。スキル一覧表に依存関係列を追加する。

---

### 批判5: CLAUDE.md の扱いが TBD のまま、Why との接続が切れている

**レビュー結果**: Why の「Claude Code が文脈を掴むまでに時間がかかる」という課題を解消する主要な手段は CLAUDE.md の内容設計であるはずだが、この部分だけが「PoC 完了後に決定する（TBD）」とされている。TBD のまま放置すると PoC をいつ・何を確認したら完了とみなせるかが不明確であり、計画として機能しない。How-3 にテスト条件が書かれているのに、確認する手順が How に存在しない矛盾もある。

**改善案**: PoC の完了条件（確認すべき項目・合格基準）と、完了後に計画のどの部分を更新するかを明記する。または `@import` に依存しない最小限の CLAUDE.md 構成をデフォルト案として先に定義し、段階的に改善する方針にする。

---

### 批判6: 「Claude Code の出力先」テストが自動検証できない

**レビュー結果**: 「Claude が編集するファイルが `ideas/ideas.md` のみ」という PASS 条件は、人間がチャット操作をして Claude の振る舞いを観察しなければならない。再現性がなく、テンプレートを更新するたびに毎回手動確認が必要になる。自動化できる検証と手動確認が必要な検証が混在しているが、区別されていない。

**改善案**: 「手動確認チェックリスト」として明確に区別し、自動化できる検証（ファイルが存在するか、行数が閾値以下か）と分離する。手動確認については「結果をどこに記録するか」を定義する。

---

### 批判7: `/decision-add` のエラー処理が矛盾している

**レビュー結果**: 「`decisions/` が存在しない場合は作成してから続行」と書かれているが、テンプレート展開後なら必ず存在するはずである。一方、本当に起きうるエラー（連番が競合する、ADR のフォーマット必須項目が未記入のまま保存しようとする）については何も記載がない。

**改善案**: エラー処理を「現実に発生しうるケース」に絞り直す。具体的には「同一連番のファイルが存在する場合の採番ロジック」と「必須項目（背景・判断・理由）が未記入の場合の振る舞い」を定義する。
