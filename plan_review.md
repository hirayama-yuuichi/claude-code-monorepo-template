plan.md への批判                               
                                                                                                                                
  1. How-1 の構造的矛盾が未解決                                                                                                 
                                                                                                                                
  「root/ を GitHub テンプレートリポジトリとして登録」と書いてあるが、GitHub テンプレートリポジトリは                           
  リポジトリ全体がコピーされる仕組みで、root/ だけを選択的に配布する機能はない。結果として新規プロジェクトに root/              
  というフォルダが生まれるだけになる。「テンプレートに含めるもの: root/                                                         
  配下の全ファイル」という記述もこの矛盾を認識していない。この計画の根幹が破綻したままで、下流の How-4
  のテスト条件まで全部ずれている。       

  2. How-2 が How-4 と重複

  How-2「テンプレート更新の伝搬（同期スクリプト方式）」は実質 /template-sync スキルのことで、そのスキルは How-4                 
  にも列挙されている。同じ内容が2つのセクションに分かれており、どちらが正であるかが不明。
                                                                                                                                
  3. PoC 前提の設計を先に書いている                                                                                             
                                         
  How-3 に「⚠️  @import の動作を PoC で確認してから確定する」とある。それなのに How-3 のテスト表に「@import 動作確認（PoC）」の  
  PASS 条件まで書いている。PoC が先行条件なら、その結果が出るまで How-3 全体を書けないはず。設計と検証の順序が逆。
                                                                                                                                
  4. テスト基準に非公式ファイルを参照している                                                                                   
                                         
  テスト PASS 条件が「モノレポ案.md の仕様と差分ゼロ」という表現を複数回使っている。モノレポ案.md                               
  はルートに置かれた非公式メモであり、正式な仕様書として扱うには不適切。root/ の実ファイルと比較するか、モノレポ案.md
  を正式な仕様書に昇格させるかを決めるべき。                                                                                    
                                                                                                   
  5. 計画ファイルが3つに分裂している     

  ls を見ると plan.md、plan_review.md、plan_変更案.md の3ファイルが存在する。プロジェクトの Why                                 
  は「製造物の均一性・見やすさ」なのに、その計画書自体が分散している。プロジェクトが掲げる価値観と真逆の状態。
                                                                                                                                
  6. root/CLAUDE.md も docs/setup.md も空のまま                                                                                 
                                         
  How-3 で「root/CLAUDE.md はエントリポイントのみ」、How-4 で「インストール手順は docs/setup.md                                 
  に記載」とあるが、両ファイルとも存在しないかプレースホルダーのまま。参照先が機能していない。

  核心： How-1 の設計ミスを直さないと、残りの How-2〜4 はすべて無効になる。そこを先に決めるべき。
