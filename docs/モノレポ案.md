rootフォルダ配下に以下方針でモノレポの雛形をつくった。
最終的には
モノレポ生成
package生成

モノレポ（信念：簡素化を貫く）
root/
│
├── README.md
│   # リポジトリの入口
│   # ・何のプロジェクトか
│   # ・packages の一覧と役割
│   # ・最短の使い方
│
├── docs/
│   # 「読むための情報」はここに集約（原則ここ以外に書かない）
│   # 時間で変わらない or 変化が遅いものだけ置く
│
│   ├── setup.md
│   │   # 環境構築手順（新人がここだけ見れば動く状態まで行ける）
│   ├── architecture.md
│   │   # 全体構成
│   │   # ・パッケージ間の依存関係
│   │   # ・責務の分離
│   │   # ・パッケージ一覧と位置づけ
│   └── decisions/
│       # 設計判断ログ（超重要・必ず残す）
│       # 「なぜそうしたか」を記録
│       └── 0001-use-xxx.md
│
├── tasks/
│   # 「動く情報」はここに集約（時間で変わるもの）
│   # 状態ごとに分けるが、役割は最小限にする
│
│   ├── backlog.md
│   │   # やること一覧（唯一のTODO）
│   │   # 思いついたらここに書く（他は禁止）
│   ├── plan.md
│   │   # プロジェクト全体の計画（パッケージ横断）
│   │   # ・マイルストーン
│   │   # ・パッケージ間の依存・順序
│   │   # ・優先順位の根拠
│   │   # パッケージ内部の詳細は書かない（個別planへ）
│   └── backlog_yyyymmdd.md
│       # 完了ログ
│
├── ideas/
│   # アイデアはここ一択（分散禁止）
│   # 「まだやるかわからないもの」専用
│   # スコープはファイルではなく見出しで管理する
│
│   └── ideas.md
│       # 例：
│       # ## Global
│       # ## package-a
│       # ## package-b
│
└── packages/
    # 実装はすべてここ
    # 「コード + そのパッケージ固有の情報」に閉じる
    #
    ├── package-a/
    │   ├── README.md
    │   │   # このパッケージの入口（必須）
    │   │   # ・何をするものか
    │   │   # ・使い方
    │   │   # ・依存関係
    │   │
    │   ├── setup.md
    │   │   # このパッケージ単体の開発手順
    │   │
    │   ├── plan.md
    │   │   # このパッケージ内に閉じた計画
    │   │   # ・パッケージ内の実装フェーズ
    │   │   # ・内部の優先順位と根拠
    │   │   # ・スパイク・検証事項
    │   │   # 横断の話は tasks/plan.md へ
    │   │
    │   └── spec/
    │       # 仕様は「分割前提」（肥大化防止）
    │       # spec.md は作らない
    │
    │       ├── api.md
    │       │   # API仕様
    │       │
    │       ├── data.md
    │       │   # データ構造・スキーマ
    │       │
    │       └── rules.md
    │           # ビジネスルール・制約
    │
    └── package-b/
        └── ...
