# 設計判断ログ

## 0002: このリポジトリ自体のパッケージ構成

- 日付: 2026-04-25
- 状態: 採用

### 背景

このリポジトリ（`p030_claude_code_base`）自体が、配布する `root/` のテンプレート構造に沿っていない。現状では `tasks/`・`ideas/`・`docs/` がトップレベルに存在せず、プロジェクトが掲げる「均一な構造」と矛盾する。

また、Claude Code スキルのソースファイル（SKILL.md 等）が `~/.claude/skills/imported/` にしか存在せず、このリポジトリで一元管理されていない。

### 判断候補

このリポジトリ自身も `root/` のテンプレート構造を採用し、以下のパッケージ構成とする。

| パッケージ | パス | 役割 |
|---|---|---|
| `template` | `packages/template/` | 配布するモノレポテンプレートの実体 |
| `skills` | `packages/skills/` | Claude Code スキルのソースファイル群 |

#### `packages/skills/` の役割

- スキルのソース（`SKILL.md` + `.provenance.json`）をこのリポジトリで一元管理する
- 管理対象スキル: `critique`、`propose`、`decision-add`、`monorepo-init`、`package-add`、`template-sync`
- デプロイ先: `.claude/skills/<スキル名>/`（プロジェクトローカル）
- デプロイ手順は `packages/skills/plan.md` に記載する

### 理由

- このリポジトリ自体がテンプレートの実例になることで、設計の整合性が保てる
- スキルをリポジトリで管理することで、変更履歴・レビュー・同期が git ベースで行える

### 決定事項

#### 1. このリポジトリ自体もテンプレート構造を採用する

`packages/template/` と同じディレクトリ構成（`tasks/`・`ideas/`・`docs/`）をトップレベルに置く。

#### 2. `root/` の置き場所 → `packages/template/` に移動

`packages/skills/` と同じ階層でパッケージを管理する。`degit` の配布 URL は `hirayama-yuuichi/claude-code-monorepo-template/packages/template` に変更する。

#### 2. スキルのデプロイ方法 → シェルスクリプト（`packages/skills/deploy.sh`）

手動コピーはミスが起きやすく、スキルとして実装すると「スキルをデプロイするためにスキルが必要」という循環になる。シェルスクリプトなら git 管理でき、1コマンドで完結する。

### 却下した選択肢

| 案 | 却下理由 |
|---|---|
| このリポジトリの構造は現状維持 | テンプレートと矛盾したまま運用することになる |
| スキルは `~/.claude/` にのみ置く | 変更履歴が残らず、複数環境への展開もできない |
