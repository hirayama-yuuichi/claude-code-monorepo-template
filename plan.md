# 全体計画
> パッケージ横断の話だけ書く。個別の実装詳細は `packages/<pkg>/plan.md` へ。
モノレポのテンプレートを各プロジェクトで共通に利用する仕組みがほしい

## Why（なぜ作るか）
> ⚠️ 凍結。変更はせず、下の How に書く。
横断的に製造物に対して、みやすさを求めたい
みやすさの向上には
　製造物間での均一性が重要と考える
そのため、モノレポの作成基本として
　モノレポ案.md
　root配下「モノレポ」のサンプルを作った。


## What  ゴール（何を達成するか）

今後はこれを各プロジェクトで利用していきたい
また、簡単に各プロジェクトで利用する方法も考えたい

## How（どうやって実現するか）

### 1. テンプレートリポジトリ（構造の配布）
- `root/` を GitHub テンプレートリポジトリとして登録
- 新規プロジェクトは `gh repo create --template <このリポジトリ>` で生成

### 2. CLAUDE.md 分割（サイズ節約）
- `root/CLAUDE.md` は薄いエントリポイント（10行以内）のみ
- 詳細ルールは `@docs/claude/workflow.md` 等に分割して `@import`
- パッケージ固有ルールは `packages/*/CLAUDE.md` に委譲

### 3. Claude Code スキルで操作自動化
- `/monorepo-init` スキル: 新規モノレポの構造を対話的に生成
- `/package-add` スキル: パッケージをテンプレートからコピーして追加
- グローバルスキル（`~/.claude/skills/`）として全プロジェクトで共用

## テスト方法

### How-1: テンプレートリポジトリ
- `gh repo create --template` で新規リポジトリを作成し、`root/` の構造がそのまま反映されることを確認

### How-2: CLAUDE.md 分割
- `root/` で `claude` を起動し「このリポジトリの構成を教えて」と聞いて、パッケージ構成・ワークフローを正しく答えられるか確認
- 「タスクを追加したい」と聞いて `tasks/backlog.md` に誘導されるか確認

### How-3: スキルによる操作
- `/monorepo-init` を実行し、`root/` と同じ構造のフォルダが生成されることを確認
- `/package-add` を実行し、`packages/<新パッケージ>/` が正しいテンプレートで生成されることを確認

### Claude Codeの出力が正しい場所に反映されるか
| 出力の種類 | 正しい場所 |
|---|---|
| 新規アイデア | `ideas/ideas.md` |
| やること追加 | `tasks/backlog.md` |
| 横断的な計画変更 | `tasks/plan.md` |
| パッケージ固有の計画 | `packages/<pkg>/plan.md` |
| 設計判断ログ | `docs/decisions/XXXX-*.md` |
| アーキテクチャ変更 | `docs/architecture.md` |

### CLAUDE.md サイズ確認
- ルートの `CLAUDE.md` が10行以内に収まっていることを目視確認

### @import の動作確認
- `root/` で `claude` を起動し、`docs/claude/*.md` の内容（ワークフロールール等）について問いかけ、正しく答えられることを確認

### 別プロジェクトでのスキル動作
- 全く別のリポジトリで `/monorepo-init` を実行し、`root/` と同じ構造が生成されることを確認

### ネガティブテスト
- 「新しいアイデアがある」と言ったとき `tasks/backlog.md` ではなく `ideas/ideas.md` に誘導されることを確認
- 「やることを追加したい」と言ったとき `ideas/ideas.md` ではなく `tasks/backlog.md` に誘導されることを確認

### 既存プロジェクトへの適用（安全性）
- 既存リポジトリで `/monorepo-init` を実行したとき、既存ファイルを上書きせずスキップまたは警告することを確認
