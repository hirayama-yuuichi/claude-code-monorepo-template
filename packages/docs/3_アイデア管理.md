# アイデア管理

`in-progress/` に置くファイルの役割と、アイデアが実装まで辿り着くフローを説明する。

## in-progress/ の全体像

| ファイル | 役割 | 管理主体 |
|---|---|---|
| `plan/` | パッケージ横断の計画（Why/What/How） | 人間・Claude |
| `ideas.md` | まだやるか不明なアイデアの一元管理 | 人間 |
| `records/` | 設計判断ログ。採用後もステータスで追跡 | Claude |
| `backlog.md` | 作業リスト（優先度順） | 人間 |

## ideas.md の書き方

思いついたアイデアはすべて `in-progress/ideas.md` に書く。
「まだやるか不明なもの」専用の置き場で、ここ以外に書かない。

- **書くもの**：やるかどうか未決定のアイデア。見出し1行と簡単な説明のみ
- **書かないもの**：「やると決めたもの」は backlog.md へ、「詳細な検討内容」は records へ

## アイデアが実装まで辿り着くフロー

```
plan/ に全体の計画を作る
    ↓
必要があれば ideas.md にアイデアを書く
    ↓ やると決まったら
/records-add で records 化 → backlog.md にも追加
    ↓
実装・docs 更新 → record のステータスを完了に更新
    ↓
backlog.md も済にする
```

records のステータスフィールド：

```markdown
- 状態: 提案 | 議論中 | 採用 | 廃止
- ドキュメント: 未着手 | 作業中 | 完了
- 実装: 未着手 | 作業中 | 完了
```

## 判断に迷うときの採用ループ

records の内容や設計方針に迷いがあるときは、批判と修正案で迷いを取り除く：

```
/critique <ファイルパス>   → _review.md を生成
    ↓
/propose <ファイルパス>    → _review.md をもとに _proposal.md を生成
    ↓
比較・採用（元ファイルを proposal で更新し、_review.md と _proposal.md を _old/ へ移動）
```

## 使用できるスキル

| スキル | 用途 |
|---|---|
| `/records-add` | 設計判断ログを新規作成 |
| `/critique` | ファイルを批判し `_review.md` を生成 |
| `/propose` | `_review.md` をもとに修正案を生成 |
